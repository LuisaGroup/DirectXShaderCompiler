# DXC Project Target Summaries

## HLSL/DXIL Core

The HLSL/DXIL Core comprises three interdependent libraries that form the heart of the DirectX Shader Compiler: `lib/DXIL` defines the DXIL data model and metadata serialization, `lib/HLSL` implements the transformation pipeline that lowers high-level HLSL IR into valid DXIL, and `lib/DxcSupport` provides foundational compiler utilities. `lib/DXIL` is anchored by `DxilModule`—the canonical representation of a shader module that tracks resources (SRVs, UAVs, constant buffers, samplers), signatures, shader model versions, and subobjects—and by `OP`, which manages the DXIL operation tables and intrinsic function cache. `DxilMetadataHelper` bridges these in-memory structures to LLVM named metadata nodes, enabling round-trip serialization. `lib/HLSL` acts as the transformation engine: `HLModule` holds high-level shader state during frontend/codegen, then `DxilGenerationPass` copies everything into `DxilModule`. A suite of lowering and legalization passes—including `HLMatrixLowerPass`, `HLSignatureLower`, `HLOperationLower`, and various DXIL-specific cleanup passes—progressively reshape the IR until it conforms to DXIL rules. `lib/DxcSupport` underpins both layers with command-line option parsing (`HLSLOptions`), file I/O (`FileIOHelper`), Unicode handling, and cross-platform compatibility (`WinAdapter`).

Together these libraries implement the complete core compilation pipeline from parsed HLSL through high-level IR, DXIL generation, and metadata emission. They are the components that every other part of DXC ultimately depends upon: the frontend generates `HLModule`, the backend containerizer reads `DxilModule`, the validator checks DXIL semantics, and the API surface exposes compile options parsed by `DxcSupport`. Without these three libraries, DXC would have no IR representation, no lowering logic, and no way to interface with the host operating system.

## DXIL Infrastructure

The DXIL Infrastructure libraries manage the complete lifecycle of a compiled DXIL shader binary: packaging, validation, root signature handling, hashing, and debug metadata compression. `lib/DxilContainer` implements the DXIL container format—the binary archive that bundles bitcode, signatures, pipeline state validation (PSV0), runtime reflection (RDAT), root signatures, debug info, and source metadata into a single `DXBC`-headed blob. Key abstractions include `DxilContainerReader` for parsing, `DxcContainerBuilder` for COM-based container modification, and `SerializeDxilContainerForModule` for assembly. `lib/DxilValidation` ensures correctness through comprehensive module-level and container-level checks: `DxilValidation.cpp` verifies instructions, signatures, resources, and shader model constraints, while `DxilContainerValidation.cpp` confirms that every container part matches the module it claims to represent. `lib/DxilRootSignature` handles the DirectX 12 descriptor binding model, providing parsing, serialization, version conversion, and validation (`RootSignatureVerifier`) to detect overlapping registers and inconsistent descriptor tables.

Supporting these are `lib/DxilHash`, which computes MD5-derived retail and debug container hashes, and `lib/DxilPdbInfo`, which compresses debug metadata using miniz for embedding as `PDBI` container parts. These components are tightly layered: `DxilHash` provides integrity checking for the container builder; `DxilRootSignature` operates independently but its serialized form becomes a container part; `DxilContainer` assembles everything; and `DxilValidation` sits above all of them to enforce both semantic and structural integrity. This infrastructure is what allows DXC to produce validated, portable, and debuggable shader binaries consumable by the DirectX 12 runtime and graphics drivers.

## LLVM Core

The LLVM Core libraries in DXC are a fork of LLVM 3.7 adapted for HLSL/DXIL compilation, retaining the standard five-layer LLVM architecture: IR, Analysis, Transforms, CodeGen, and Pass management. `lib/IR` defines the in-memory program representation—`Value`, `Instruction`, `BasicBlock`, `Function`, `Module`, `Type`, `LLVMContext`, and the pass infrastructure base classes—together with SSA verification, IR printing, and debug metadata support. DXC-specific hooks such as `ResetHLModule()` and `ResetDxilModule()` attach HLSL/DXIL state to `llvm::Module`. `lib/Analysis` computes program properties consumed by optimizations: alias analysis, dominator trees, loop information, scalar evolution, memory dependence, and DXC-specific analyses like `DxilValueCache`, `DxilConstantFolding`, and `DxilSimplify` that understand DXIL opcode encoding. `lib/Transforms` contains the optimization and canonicalization passes—InstCombine, GVN, LICM, SROA, inlining, loop transforms, vectorization, and a large collection of DXC-specific passes such as `DxilLoopUnroll`, `DxilEliminateVector`, and `DxilEraseDeadRegion` that adapt IR for DXIL constraints. `lib/Passes` provides the new pass manager infrastructure (`PassBuilder`), while `lib/CodeGen` lowers IR to machine instructions via SelectionDAG, register allocation, instruction scheduling, and assembly/object emission.

These libraries are the backbone of DXC's optimizer and code generator. DXC customizations are localized but pervasive: vector elimination for DXIL scalarization, special loop unroll handling for HLSL `[unroll]`, DXIL intrinsic constant folding, and integration with `DxilModule`/`HLModule` metadata. The pass-centric design means nearly all IR manipulation is organized as passes operating on SSA-form IR, orchestrated by the legacy and new pass managers. While DXC primarily targets DXIL rather than native CPU code, the full LLVM backend machinery remains available and is used for JIT/interpreter scenarios and tooling.

## LLVM Infrastructure

The LLVM Infrastructure libraries provide the bedrock upon which the entire DXC compiler is built, handling platform abstraction, bitcode serialization, textual IR parsing, module linking, and unified IR loading. `lib/Support` is the foundational utility library with ~107 files plus platform subdirectories, providing `raw_ostream` for buffered output, `MemoryBuffer` for efficient file access, `SourceMgr`/`SMDiagnostic` for caret diagnostics, `CommandLine` for declarative option parsing, arbitrary-precision arithmetic (`APInt`/`APFloat`), containers (`SmallVector`, `StringMap`, `FoldingSet`), and cross-platform abstractions for paths, file systems, threading, and signals. DXC customizations include MS file system integration, exception-based error handling on Windows, and DXIL architecture support in `Triple`. `lib/Bitcode` handles binary serialization of LLVM IR through `BitcodeReader` and `BitcodeWriter`, with low-level bitstream decoding and value enumeration for deterministic output; DXC adds bulk array reading optimizations and DXIL operation awareness. `lib/AsmParser` converts `.ll` text into in-memory `Module` objects via `LLLexer` and `LLParser`. `lib/Linker` merges multiple modules with type remapping and symbol resolution. `lib/IRReader` auto-detects bitcode versus assembly and provides a unified loading interface.

These libraries are highly stable and extensively tested, forming the basis for all IR manipulation in DXC. HLSL-specific changes are minimal and localized—primarily around Windows file system integration, exception-based fatal error handling, and DXIL-specific extensions to the target triple and bitcode reader. Every DXC compilation path touches these components: source files are read through `MemoryBuffer`, parsed IR is loaded via `IRReader`, optimized modules are written via `BitcodeWriter`, and linked libraries are merged through the `Linker`.

## Backend & Machine Code

The Backend & Machine Code libraries bridge LLVM IR down to executable machine code and binary object formats. `lib/MC` (Machine Code) is the lower-level layer responsible for assembly and object file generation/disassembly, containing `MCInst`/`MCOperand` for instruction representation, `MCStreamer` with subclasses for text assembly (`MCAsmStreamer`) and binary objects (`MCObjectStreamer`), `MCAssembler` for fragment-based layout and fixup resolution, `MCContext` for symbol/section ownership, format-specific object writers (ELF, Mach-O, COFF), and a disassembly framework (`MCDisassembler`). `lib/Target` provides the target-independent code generation interface through `TargetMachine`, which holds the data layout, target triple, and CPU features, and drives the entire codegen pipeline via `addPassesToEmitFile`. `TargetLoweringObjectFile` assigns IR globals to object sections, while `TargetSubtargetInfo` tracks per-function ISA features.

`lib/LTO` implements link-time optimization through `LTOCodeGenerator` and `LTOModule`, merging bitcode modules, applying interprocedural optimizations, and emitting native object files. `lib/ExecutionEngine` provides JIT compilation and direct IR execution via `MCJIT`, `OrcMCJITReplacement`, `RuntimeDyld`, and an LLVM IR `Interpreter`. In DXC's shader compilation path, the MC layer is heavily used for DXIL validation and container object emission, while the Target layer provides the abstraction for the HLSL compiler backend. LTO and ExecutionEngine are less central to the main DXIL pipeline—DXC primarily compiles HLSL to DXIL bytecode rather than native machine code or JIT execution—but they remain part of the inherited LLVM infrastructure and are used by tooling and testing scenarios.

## Debug, PIX & DIA

The Debug, PIX & DIA directories provide shader debugging, profiling, and instrumentation capabilities that are essential for developer tooling. `lib/DxilDia` implements Microsoft's Debug Interface Access (DIA) API for DXIL modules, enabling tools like PIX and Visual Studio to inspect HLSL debug metadata, source locations, symbols, variables, and types. Its core classes include `dxil_dia::DataSource` (`IDiaDataSource`) for loading DXIL/PDB streams, `dxil_dia::Session` (`IDiaSession`) for maintaining instruction-to-RVA maps and line info, and `dxil_dia::Symbol` (`IDiaSymbol`) for exposing debug symbol properties. PIX-specific interfaces such as `IDxcPixDxilDebugInfo`, `IDxcPixDxilStorage`, and `IDxcPixVariable` map HLSL variables to DXIL registers and answer queries about live variables at instruction offsets. `lib/DxilPIXPasses` contains LLVM instrumentation passes that modify shaders at compile time to enable PIX analysis: `DxilDebugInstrumentation` adds UAV-based execution tracing, `DxilAnnotateWithVirtualRegister` assigns register IDs for PIX correlation, `DxilShaderAccessTracking` records resource usage patterns, and `PixPassHelpers` provides shared infrastructure for UAV creation and root signature extension.

`lib/DxrFallback` implements the DXR Fallback Compiler, which transforms DXR (DirectX Raytracing) shader libraries into a single compute shader for hardware without native DXR support. Its `StateFunctionTransform` splits functions at `TraceRay`/`CallShader` calls into explicit state-machine substates with software-managed stack frames, while `LiveValues` computes liveness across continuation points. `lib/DxilCompression` provides zlib compression/decompression via miniz for debug info payloads. `lib/DebugInfo` contains LLVM's standard DWARF and PDB parsers, with the DIA subsystem (`DebugInfo/PDB/DIA`) wrapping Microsoft's DIA SDK for cross-platform PDB access. Together these components enable comprehensive shader debugging and profiling: `DxilPIXPasses` instruments at compile time, the GPU executes instrumented shaders writing trace data to UAVs, and `DxilDia` resolves that data back to HLSL source during debug sessions.

## Clang Frontend

The Clang Frontend in DXC is a deeply modified fork of LLVM/Clang where HLSL support is woven throughout every major subsystem. `lib/Lex` adds HLSL keywords and `HLSLMacroExpander` for post-lexing semantic macro expansion. `lib/Parse` extends the parser with `ParseHLSL` for `cbuffer`/`tbuffer` declarations, HLSL `[attribute]` syntax, and root signature sub-language parsing (`HLSLRootSignature`). `lib/AST` contains HLSL-specific type system extensions in `HlslTypes.h`—scalar types, swizzle/member access tracking, and `UnusualAnnotation` hierarchies for register assignments, semantics, and payload access qualifiers—plus `HlslBuiltinTypeDeclBuilder` for declaring builtin object types like `Texture2D` and `RWBuffer`. `lib/Sema` is the largest HLSL customization: `SemaHLSL.cpp` (~18K lines) handles type checking, overload resolution, vector/matrix operations, implicit conversions, template deduction, intrinsic tables, and attribute processing, while `SemaHLSLDiagnoseTU.cpp` performs translation-unit-level validation including recursion checking, entry point validation, patch constant function verification, and library export checks.

`lib/CodeGen` is dominated by `CGHLSLMS.cpp` (~268K lines) and `CGHLSLMSFinishCodeGen.cpp` (~156K lines), which together generate high-level DXIL (HLModule) from the validated AST—handling resources, entry function signatures, matrix/vector codegen, init list flattening, root signature emission, subobject creation, and control flow structurization. `lib/SPIRV` provides an alternative backend for Vulkan targets. The tools subdirectory adds DXC-specific binaries: `dxc.exe`, `dxcompiler.dll` (implementing `IDxcCompiler`), validators, assemblers, linkers, and the DXR fallback compiler. HLSL modifications are explicitly marked with `// HLSL Change` comments, and key architectural decisions include disabling modules/PCH/AST serialization, forbidding recursion, treating arrays as value types (no pointer decay), and emitting high-level DXIL that is later lowered by DXC backend passes.

## Tools

The `tools/` directory (excluding Clang) contains a rich ecosystem of 38+ tool directories inherited from LLVM, with DXC-specific enhancements and one entirely new tool. `dxexp` is the sole DXC-native tool; it enumerates DXGI adapters and queries D3D12 experimental feature support including shader model tiers, wave operations, raytracing, mesh shaders, and wave matrix capabilities, reporting results in text or JSON. Core LLVM IR tools include `llc` (native code generation), `llvm-as` (assembly to bitcode), `llvm-dis` (bitcode to assembly, enhanced to extract DXIL from DXBC containers), `opt` (optimization pass driver with HLSL/DXIL pass integration), and `llvm-link` (module linker). HLSL modifications to these tools typically involve `__cdecl` calling conventions, Windows file system integration, linking against DXC libraries, and disabling irrelevant LLVM components like LTO or ObjCARCOpts.

Object file tools (`llvm-mc`, `llvm-nm`, `llvm-objdump`, `llvm-readobj`, `llvm-size`, `macho-dump`, `llvm-cxxdump`) provide assembly, disassembly, and object analysis. Debug info tools (`dsymutil`, `llvm-dwarfdump`, `llvm-pdbdump`, `llvm-symbolizer`) support DWARF and PDB formats. Bitcode analysis tools (`llvm-bcanalyzer`, `llvm-diff`, `llvm-extract`) enable introspection and comparison of IR modules. Testing utilities (`llvm-stress`, `verify-uselistorder`, `llvm-c-test`) support compiler fuzzing and serialization validation. Build tools (`llvm-config`, `msbuild`) provide configuration reporting and Visual Studio integration. LTO tools (`llvm-lto`, `lto`) support link-time optimization, and conversion tools (`obj2yaml`, `yaml2obj`) help construct precise test cases. Together these tools support DXC development, testing, and deployment by providing IR manipulation, object inspection, debug analysis, and build integration capabilities.

## Include Headers

The include directories expose DXC's layered API surface, spanning public COM interfaces, LLVM IR infrastructure, DXIL representation, and stable C bindings. `include/dxc` is the primary public API: `dxcapi.h` defines `DxcCreateInstance` and the core compiler interfaces (`IDxcCompiler3`, `IDxcUtils`, `IDxcBlob`, `IDxcResult`), while sibling headers provide tooling (`dxctools.h`), IntelliSense (`dxcisense.h`), PIX debugging (`dxcpix.h`), DXR fallback compilation (`dxcdxrfallbackcompiler.h`), and cross-platform COM compatibility (`WinAdapter.h`). Subdirectories `include/dxc/DXIL` (30 headers) define the DXIL intermediate representation—`DxilModule`, `OP`, `DxilShaderModel`, signatures, resources, type system, metadata helpers, and instructions. `include/dxc/HLSL` (25 headers) covers high-level IR (`HLModule`), lowering passes, matrix handling, signature allocation, and ViewID pipeline validation. `include/dxc/Support` (24 headers) contains option parsing, file I/O, language extensions, and Windows portability shims.

`include/llvm` houses the LLVM 3.7 compiler infrastructure headers used internally by DXC, with the `IR` subdirectory (88 headers) defining `Module`, `Function`, `Value`, `Instruction`, `Type`, `IRBuilder`, `LLVMContext`, metadata, and pass management. Other subdirectories cover bitcode, support utilities, transforms, analysis, code generation, MC, target backends, and debug info formats. `include/llvm-c` provides a stable C API with opaque types (`LLVMModuleRef`, `LLVMValueRef`, etc.) for tooling and language bindings. `include/miniz` contains the single-header zlib replacement used for PDB compression. This layered header structure reveals DXC's architecture: a stable COM API at the surface, LLVM IR as the universal intermediate language, DXIL as the shader-specific dialect, and HLSL frontend passes bridging parsed source to low-level shader representation.

## Testing

DXC's testing infrastructure is a multi-layered system combining LLVM's LIT framework, GoogleTest for C++ unit tests, and Microsoft's TAEF for Windows-native integration testing. The LLVM core regression tests in `test/` are mostly disabled for HLSL via `lit.local.cfg` files, with only curated subsets active—FileCheck, TableGen, Verifier, LTO, and a small `test/HLSL/` directory. The `unittests/` directory contains 131 GoogleTest `.cpp` files covering LLVM ADT, Analysis, IR, Bitcode, Linker, Support, Transforms, AsmParser, plus DXC-specific tests like `DxcSupport/WinAdapterTest.cpp` and `DxilHash/DxilHashTest.cpp`. Excluded directories include ExecutionEngine, LineEditor, and most CodeGen/MC unit tests because HLSL does not target CPU architectures.

The most active HLSL/DXIL test area is `tools/clang/test/`, which contains thousands of tests across multiple categories: `SemaHLSL/` (~261 files) for semantic analysis, `CodeGenDXIL/` (~135 files) and `CodeGenHLSL/` (~105 files) for code generation, `HLSLFileCheck/` (~2,210 files, TAEF-driven) for broad regression coverage, `HLSLFileCheckLit/` (~24 files) for lit-native HLSL tests, `DXC/` (~153 files) for driver tests, and `LitDXILValidation/` (~35 files) for DXIL validator tests. SPIR-V backend tests in `CodeGenSPIRV/` comprise ~1,573 files, reflecting first-class Vulkan support. TAEF runs the bulk of HLSL tests via `ClangHLSLTests.dll` on Windows, while lit-enabled suites provide cross-platform validation. DXIL backward compatibility targets test against released `dxil.dll` validators. The `projects/dxilconv/test/` directory contains DXBC-to-DXIL converter tests. Overall, DXC maintains ~9,000+ test files with ~2,800+ HLSL/DXIL-specific tests, ensuring correctness across semantic analysis, code generation, validation, reflection, and SPIR-V emission.

## Documentation

The `docs/` directory is a hybrid set containing upstream LLVM documentation inherited from the fork and DXC-specific specifications for HLSL, DXIL, and SPIR-V. The core DXC documents are `HLSLChanges.rst`, which explains the architectural rationale for forking LLVM/Clang and details dependency injection via `MSFileSystem`, COM-lite component design, C++ exception/HR`ESULT error handling, and the HLModule/DxilModule dual-module system; `DXIL.rst`, the authoritative specification for the DirectX Intermediate Language covering versioning, type system, memory model (LLVM address spaces), reducible control flow requirements, shader signatures, resources, and the full DXIL instruction set; `SPIR-V.rst`, a comprehensive HLSL-to-SPIR-V mapping manual covering entry wrappers, descriptor bindings, Vulkan attributes, type mappings, buffer types, system value semantics, and legalization rules; and `SPIRV-Cookbook.rst`, a practical guide showing accepted and rejected HLSL patterns for SPIR-V compilation.

Additional DXC-specific docs include `SourceLevelDebuggingHLSL.rst` for debug info container parts and command-line options, `BuildingAndTestingDXC.rst` for CMake build instructions and LIT/TAEF testing, and `ReleaseNotes.md` tracking releases from 1.7.2207 through 1.9.2602 and experimental Shader Model 6.10 features. Inherited LLVM documentation covers `LangRef.rst`, `CodingStandards.rst`, `ProgrammersManual.rst`, `Passes.rst`, `BitCodeFormat.rst`, `CodeGenerator.rst`, and more. Build infrastructure includes Sphinx configuration (`conf.py`), Makefile support, and a custom DXC theme (`_themes/dxc-theme/`). This documentation set serves distinct audiences: `DXIL.rst` for driver/compiler backend developers, `SPIR-V.rst` and `SPIRV-Cookbook.rst` for Vulkan shader developers, `HLSLChanges.rst` for compiler integrators, and `BuildingAndTestingDXC.rst` for contributors building from source.

## Build System & Utilities

DXC's build system is a sophisticated extension of LLVM's CMake infrastructure, heavily customized for HLSL/DXIL and SPIR-V compilation. The `cmake/` directory provides modules for compiler detection, platform abstraction, Windows SDK/D3D12/TAEF discovery, and DXC-specific code generation. Key modules include `HandleLLVMOptions.cmake` for compiler version validation and EH/RTTI flags, `HCT.cmake` for the `add_hlsl_hctgen()` function that generates source from `hctgen.py`, `TableGen.cmake` for `.td` file processing, `FindD3D12.cmake` and `FindDiaSDK.cmake` for Windows SDK components, and `FindTAEF.cmake` for the Test Authoring and Execution Framework. `cmake/caches/PredefinedParams.cmake` provides pre-configured cache variables for *nix builds, targeting `dxil-ms-dx` with no native backends, SPIR-V codegen enabled, and HLSL tests included.

The `utils/` directory contains LLVM legacy utilities plus DXC-specific build/test automation. The `utils/hct/` suite (`hctstart`, `hctbuild`, `hcttest`) provides a polished Windows command-line workflow managing environment setup, CMake configuration, building, TAEF/LIT testing, and version generation. `hctgen.py` generates C++ source files from `gen_intrin_main.txt` and the DXIL database (`hctdb.py`), producing headers and tables for intrinsics, DXIL constants, instructions, shader models, validation rules, and options. `utils/version/gen_version.py` manages three versioning schemes (official, dev, fixed) from Git metadata. Other utilities include `TableGen` for code generation from `.td` files, `FileCheck` for test pattern matching, `lit` for testing, `llvm-config` for build configuration, and numerous helper scripts for code formatting, coverage, release engineering, and finding optimization differences. The build system supports cross-compilation to Android and iOS, enforces `_ITERATOR_DEBUG_LEVEL=0` consistency, and integrates both modern CMake/LIT and legacy Windows HCT/TAEF paths.

## External Dependencies

DXC depends on four key external dependencies managed under the `external/` directory (or referenced from `utils/unittest/` for googletest). `DirectX-Headers` (MIT license, v1.4.9) provides official D3D12 headers under an MIT license with WSL compatibility shims, enabling reflection support on non-Windows platforms without the full Windows SDK. `SPIRV-Headers` (MIT license, v1.5.5) contains the machine-readable SPIR-V registry headers (`spirv.hpp`, `spirv.h`) required for SPIR-V code generation. `SPIRV-Tools` (Apache 2.0) provides assembler, disassembler, validator, and optimizer libraries (`SPIRV-Tools-static`, `SPIRV-Tools-opt`) used by the `clangSPIRV` library and DXC tools; DXC skips building executables and configures the library with `_ITERATOR_DEBUG_LEVEL=0` consistency. `Google Test` (BSD 3-Clause, in `utils/unittest/`) provides the C++ unit testing framework used by LLVM, Clang, HLSL, and SPIR-V unit tests, configured with `GTEST_HAS_RTTI=0` to match LLVM conventions.

SPIR-V support is conditional: enabled by default on Linux, optional on Windows, and pulling in both SPIRV-Headers and SPIRV-Tools when active. DirectX-Headers is required on Linux/WSL but optional on Windows where the Windows SDK is available. The root `CMakeLists.txt` and `external/CMakeLists.txt` wire these dependencies into the build, setting include directories, library links, and target properties. `tools/clang/lib/SPIRV/`, `tools/clang/tools/dxcompiler/`, `tools/clang/tools/dxclib/`, and `tools/clang/unittests/SPIRV/` all consume these dependencies. This external dependency strategy keeps DXC self-contained for its core DXIL path while enabling optional but robust SPIR-V and cross-platform DirectX support through well-versioned, actively maintained upstream projects.

## Miscellaneous Libraries & Resources

The miscellaneous components are a mix of upstream LLVM libraries adapted for DXC and Microsoft-specific additions with no upstream equivalent. `lib/Fuzzer` is LLVM's libFuzzer—a coverage-guided fuzzing engine used to test the HLSL parser and compiler front-end through mutation, corpus management, and crash detection. `lib/MSSupport` is a Microsoft addition providing a Windows file system abstraction (`MSFileSystemImpl.cpp`) that wraps Win32 APIs for consistent Unicode and long-path handling. `lib/Object` provides cross-platform object file I/O for COFF, ELF, Mach-O, and archives; in DXC it enables reading DXIL containers and treating LLVM IR as object files via `IRObjectFile.cpp`, with HLSL-specific simplifications removing unnecessary MC/MCParser dependencies. `lib/Option` is LLVM's command-line argument parsing framework, powering `dxc.exe` and related tools with typed option tables generated from TableGen descriptions. `lib/ProfileData` supports profile-guided optimization and code coverage, used primarily for DXC self-coverage builds. `lib/TableGen` is LLVM's code generator generator, used at build time to produce HLSL intrinsic tables, diagnostic tables, pass registration boilerplate, and option tables from `.td` files. `lib/PassPrinters` provides analysis printer passes for IR inspection via `opt -analyze`.

`examples/` preserves the `ModuleMaker` educational sample demonstrating LLVM API usage. `projects/` houses Microsoft-specific extensions: `dxilconv`, a complete DXBC-to-DXIL converter with COM API (`IDxbcConverter`), command-line tools, custom LLVM passes for control flow restructuring, and extensive regression tests; and Windows ETW tracing infrastructure (`DxcRuntime.man`). `resources/` contains a Windows VERSIONINFO resource template for embedding version metadata into DXC binaries. Throughout these libraries, HLSL-specific adaptations strip out irrelevant CPU backend features (native assembly parsing, object dependencies) while preserving essential infrastructure for shader compilation, testing, and Windows ecosystem integration.

## Cross-Cutting Themes

**LLVM Fork with HLSL Specialization.** DXC is fundamentally a fork of LLVM 3.7 and Clang, but it is not a shallow wrapper. HLSL support is woven throughout every layer—from lexer keywords and AST type annotations to semantic analysis rules, code generation, and backend passes. Changes are explicitly marked with `// HLSL Change` comments, creating a disciplined but extensive set of modifications that adapt a general-purpose compiler infrastructure for GPU shader compilation.

**Two-Module IR Architecture.** A distinctive DXC design pattern is the dual-module system: `HLModule` captures high-level HLSL concepts (matrices, vectors, intrinsics, resources) during frontend/codegen, and `DxilGenerationPass` transforms it into `DxilModule`, which holds the canonical low-level DXIL representation required by drivers and validators. This separation allows the frontend to work with familiar HLSL abstractions while the backend enforces strict DXIL constraints.

**Container-Centric Binary Format.** Unlike a typical compiler that emits standalone object files, DXC packages compiled output into the DXIL container—a structured archive with FourCC-identified parts for bitcode, signatures, pipeline state validation, runtime reflection, root signatures, debug info, and source metadata. This container format is the contract between the compiler, the DirectX runtime, and graphics drivers.

**Cross-Platform with Windows Roots.** While DXC builds and runs on Linux and is used for Vulkan SPIR-V generation, its origins and primary deployment target are Windows. This manifests in COM-based APIs (`IDxcCompiler`, `IDxcUtils`), Windows file system abstractions (`MSFileSystem`), ETW tracing, DIA debug interfaces, TAEF testing, and extensive Windows SDK integration. Cross-platform support is achieved through adapters like `WinAdapter.h` and DirectX-Headers rather than by removing Windows dependencies.

**Generated Code and Specification Databases.** A significant portion of DXC's source code is generated from declarative definitions rather than hand-written. `hctgen.py` consumes `gen_intrin_main.txt` and `hctdb.py` to generate intrinsic tables, DXIL constants, instruction definitions, shader model headers, validation rules, and option parsers. TableGen similarly generates pass registration, option tables, and diagnostic messages from `.td` files. This approach ensures consistency across the compiler's many tables and reduces manual maintenance burden.

**Validation at Every Layer.** DXC embeds validation throughout the compilation pipeline: the Clang frontend enforces HLSL language rules (no recursion, no array-to-pointer decay), semantic analysis checks availability and entry points, DXIL generation passes legalize constructs, `lib/DxilValidation` verifies module and container correctness, and external `dxil.dll` validators provide additional conformance checking. This layered validation is critical because DXIL is a contract between compilers and GPU drivers where malformed input can cause runtime instability.
